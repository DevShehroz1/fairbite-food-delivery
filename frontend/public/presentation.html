The elevator pitch

One platform, four people, zero call-center.

QuickBite is a working food-delivery marketplace built for the Lahore market. Customers browse restaurants and pay on delivery, restaurants accept orders and update prep status, riders accept pickups and complete drops, and admin oversees the whole system. Everything is live — chat, order status, rider location — and synced through a Postgres backend with 2-second polling.

Why this exists

Lahore is a food city without a Lahore-first delivery app.

Existing apps treat Lahore like any other city — generic UX, English-only flows, no local restaurants on Day 1, and pricing tuned for Western markets. QuickBite is built for the way people here actually order food.

The actual tech we shipped.

The final report mentioned LocalStorage and Bootstrap — that was the Sprint-1 plan. By Sprint-3 we'd outgrown it. The shipped build is a full client-server app.

Backend

Infra & Deploy

Vercel (frontend & backend) Supabase cloud Postgres Firebase Auth (Spark/free) GitHub source of truth

Why these choices

Postgres for relational integrity + JSONB for flexible cart/pricing. Vercel for free hosting tied to GitHub. Firebase Phone Auth gave us free OTP without a paid SMS gateway. Polling beats WebSockets on serverless — simpler, same UX at 2-second cadence.

Architecture story

From LocalStorage prototype to multi-role marketplace.

Sprint 1 was a quick LocalStorage proof. Once roles multiplied, we couldn't keep state in the browser — a rider's accept needs to be visible to the customer's tracking screen instantly. So we lifted state into Postgres and kept the React UI exactly as designed.

Real-time via polling

Order status, chat, and rider location refresh every 2 s. Cheap, reliable, and Vercel-friendly.

Race-safe accepts

Rider accept = single UPDATE with rider_id IS NULL guard. Two riders can tap at once — only one wins.

Idempotent rewards

Referral credit is gated on user state, so re-runs never double-issue. Same for first-order coupons.

Anticipated questions

Likely teacher Q & A.

Click to expand. Bold sentence first, then the one-line "why" so you can hold the room.

The final report mentions LocalStorage and Bootstrap. Where did they go?

That was the Sprint-1 plan. Once we added riders and restaurants, browser storage couldn't work — a rider's accept has to be visible to the customer instantly. We lifted state into Supabase Postgres and kept the React UI we'd already designed.

This is the classic "plan vs. ship" gap that Agile is supposed to absorb.

Why polling instead of WebSockets?

The backend runs on Vercel serverless functions, which don't keep persistent connections open. Polling every 2 seconds gives the same perceived live-ness with simpler infra and no scaling surprises.

It's the right tradeoff at this scale — and the same pattern Stripe uses for dashboard updates.

How is the OTP free?

Firebase Phone Auth is free on the Spark plan when used with reCAPTCHA-verified test phone numbers. Real production SMS would need a paid plan, but for class demos the test-number flow is identical from the user's perspective.

What happens if two riders tap "accept" at the same time?

The accept is a single SQL UPDATE with a rider_id IS NULL guard. The database picks one winner atomically; the other rider gets a polite "already taken" toast.

Classic optimistic-locking pattern — no extra queues or locks needed.

Can someone abuse referral credit?

No — credit is gated on user state, not message frequency. Re-running the credit step is a no-op once both users are flagged as credited.

Why a separate admin login?

Because routing every admin email through the customer login created a hijack: anyone whose email happened to match would be funneled into the admin dashboard. Now the admin login lives at /admin/login and sends an intent: 'admin' flag — only that path checks the allowlist.

Why mobile-only for customers, but desktop for admin?

Customers order on their phone; admins work on a laptop. The customer pages run inside a 430px phone frame even on desktop. Admin pages add body.qb-fullscreen on mount, which lifts the cap globally.

Why are reviews about the rider, not the restaurant?

We added rider reviews late because the delivery quality is what customers actually want to rate — food temperature, on-time arrival, courtesy. Restaurant reviews still exist for historical records but new reviews flow to the rider's profile.

What's the database design like?

Standard relational tables for users, restaurants, orders, reviews, chats. JSONB columns where the shape is flexible — order pricing breakdown, cart items, status history, rewards. That's a Postgres-only feature we picked Supabase for.

How do you deploy?

Frontend auto-deploys on every push to master. Backend is intentionally git-disconnected on Vercel so we can push frontend changes without bouncing the API — we deploy the backend manually with vercel deploy --prod when needed.

What would you do with more time?

Real SMS gateway, push notifications, payment-gateway integration (we're cash-on-delivery only today), and an offline mode for riders with spotty connections. Plus a proper analytics pipeline instead of the lightweight in-app reports.
